## Issues — priority order

### Issue #1 — Test script uses subdomain where API expects UUID  🔥 BLOCKER

**Symptom**: PLP, PDP, search all return `0%` 2xx (every request fails). The
actual response is HTTP 405 with body `{"error":{"code":"VALIDATION_ERROR","message":"Request validation failed"}}`.

**Root cause**: The k6 script hits URLs like
`/api/v1/storefront/store/store-1/products`, putting the subdomain string
`store-1` in the `{store_id}` path slot. But the route handler types
`store_id: UUID` ([search.py:97](../../src/api/v1/routes/storefront/search.py#L97),
products route mounted at [`__init__.py:354`](../../src/api/v1/routes/__init__.py#L354)),
so FastAPI rejects the request at path-param validation before it ever runs.

The actual storefront frontend doesn't have this problem — it calls
`/api/v1/storefront/store-by-subdomain/{subdomain}` first
([public.py:451](../../src/api/v1/routes/storefront/public.py#L451)),
gets back the store UUID, and uses that UUID for all subsequent calls. The
k6 script skips the lookup step.

**Why it inflates the failure number**: every iteration makes 1 home call
+ 4 broken calls. Since the home call works, the 80% failure rate is
roughly "4 of 5 calls always fail" → 80%. Fix the script and we get a clean
read on what's actually slow vs. what was never running.

**Fix sketch** (in `scripts/load/storefront_load.js`):
1. In a per-VU `setup()` or first-iteration block, call
   `GET /api/v1/storefront/store-by-subdomain/{subdomain}` once per store
   and cache the UUID in a `__VU`-keyed lookup.
2. Use the cached UUID in every subsequent products/PDP/search URL.
3. Skip stores that 404 on the lookup (so a missing fixture doesn't poison
   the whole run).

**Effort**: ~30 min. **Impact**: should drop failure rate from 80% to
roughly the actual error rate (probably <5%).

**Pre-condition for fix to pass**: the 10 test stores (`store-1` …
`store-10`) must actually exist on the target env, and each must have been
through `/seed-demo` so `demo-tshirt` and `starter-collection` resolve.
Check with:
```bash
for i in 1 2 3 4 5; do
  curl -s "https://staging.numueg.app/api/v1/storefront/store-by-subdomain/store-$i" \
    | head -c 100
  echo
done
```

### Issue #2 — Home page p95 = 5.27 s under load  🐢 REAL PERF

**Symptom**: Home page returns 200 OK on every request (no errors) but the
95th-percentile latency is **5.27 seconds** — 13× the 400ms target. Avg
830ms, max 13.76s.

**Root cause** (hypothesis — needs profiling to confirm): the storefront's
SSR home render does several blocking server-side fetches before first
paint, all hitting the API:
- `getStoreData()` — store + theme settings
- `getActivePromotions()` — banner / popup / floating widget / cookie banner
- (potentially) cart preview, currency rates, etc.

At 100 VUs each hitting the same store(s), the Postgres connection pool
and the API process(es) on the droplet are likely contending. The fact
that p95 is 5s but median is 244ms (much faster) is the classic shape of
"works fine at low concurrency, falls over at the tail" — connection-pool
exhaustion, GC pauses, or a slow query under contention.

**Diagnostic next steps**:
1. SSH the droplet during the test window and watch `docker stats` —
   look for the api/postgres containers hitting CPU/mem limits.
2. Tail the API logs for slow-query warnings or pool-timeout errors.
3. Add a simple in-memory cache (LRU, 30s TTL) in front of `getStoreData()`
   and `getActivePromotions()` and re-run. Store data changes infrequently;
   30s of staleness is acceptable for shoppers.
4. If still slow, profile a single home request with `py-spy` or similar
   to find the hot path.

**Effort**: 2–6 hours including profiling. Likely 1–2 wins (caching +
maybe a missing index) get us under target.

### Issue #3 — Cart endpoint 51% over latency target  🐢 PERF (smaller)

**Symptom**: `/api/v1/storefront/me/cart` p95 = 2.02s vs 200ms target.
51% of requests exceed the threshold.

Note: the k6 check only measures latency, not status — cart legitimately
401s for anonymous sessions without a cookie, so the "50% pass" number is
about speed, not errors.

**Likely cause**: same as #2 — contention. Cart fetch hits cart_items +
product + variant + price joins; at 100 VUs that's a fair amount of work
per request without a cache.

**Diagnostic next steps**:
- EXPLAIN the cart query under realistic data volumes.
- Add `Cache-Control: private, max-age=5` so the browser at least doesn't
  re-fetch on every navigation.
- Consider Redis-caching the cart object for the duration of a session,
  invalidated on mutation.

**Effort**: 1–3 hours.

## Recommended order of operations

1. **Fix #1 (script)** — unblock honest measurement. Without this, you
   can't know what's actually broken.
2. **Re-run** the test against staging. Capture the real numbers.
3. **Then triage** #2 and #3 based on whatever the cleaned-up numbers show.
   If after #1 the home p95 is still 5s+, prioritize #2 hard. If the new
   numbers are within tolerance, move on.
4. **#4 (seeding)** is worth automating regardless — manual seeding is
   the kind of step that drifts (someone deletes a store, demo data goes
   stale, etc.).

## What to leave alone

- The 405-with-validation-error response shape is weird but it's downstream
  of FastAPI's UUID parsing + your error handler. Not worth changing the
  response code; the test bug is the issue, not the error response.
- The threshold values themselves (p95 < 500ms, etc.) are aggressive but
  defensible for a storefront. Don't relax them to make the test pass
  cosmetically — fix the underlying perf instead.

## Performance improvements to add

Even with the script fixed and the failure rate honest, the latencies tell
us there's real perf work waiting. Tiered by **(impact × ease)**. Start at
tier 1 — the highest leverage per hour spent. Don't bother with tier 3
until tier 1 + 2 are exhausted.

### Tier 1 — Do first (highest ROI)

These address the symptoms in #2 and #3 directly and are low-risk to
deploy.

#### 1.1 — Redis cache for store + theme + active promotions

The SSR home render does 3 blocking API fetches on every request:
`getStoreData()`, theme settings, `getActivePromotions()`. None of these
change per shopper; all change rarely (theme edits, promo activation).

- Cache by `(store_id, key)` with a 30-60s TTL in Redis (already deployed).
- Invalidate on write: store mutation endpoints already publish events —
  hook the cache eviction in the same handler.
- Estimated impact: home p95 5.27s → ~1s. Biggest single win available.
- Effort: 3-5 hours. New `src/infrastructure/cache/storefront_cache.py`
  helper plus invalidation hooks in the relevant write paths.

#### 1.2 — HTTP cache headers on read endpoints

The storefront API currently sends `Cache-Control: private, no-cache,
no-store` on most read paths (saw it in the home response headers earlier).
That's correct for authenticated routes but throws away free cache on
public read paths.

- Per-store product/category/theme/promotions endpoints can ship
  `Cache-Control: public, max-age=60, s-maxage=300, stale-while-revalidate=86400`.
- Cloudflare will serve repeats from edge; browsers will re-use within the
  TTL window. Avoids hitting the origin at all for ~60% of read traffic.
- Estimated impact: massive at the edge, near-zero load on origin for
  cached responses.
- Effort: 1-2 hours. Add a small dependency that sets the header per route.

#### 1.3 — Indexes on common storefront filter columns

EXPLAIN the slow queries (cart, products list, search) on staging with
realistic data. Likely missing:

- `(store_id, status, created_at DESC)` composite on `products` for the
  PLP order-by + filter case.
- `(store_id, status, parent_category_id)` on `categories`.
- Partial index `WHERE status = 'active'` if most queries filter to active.
- Full-text GIN index on the search columns if not already present.

Each missing index can turn a 1s query into a 5ms one under load.

- Estimated impact: highly variable but often 10-50x on the affected
  endpoint. Confirm with EXPLAIN ANALYZE first.
- Effort: 2-4 hours for analysis + alembic migration. Risk: rolling new
  indexes against a busy table briefly blocks writes — use
  `CREATE INDEX CONCURRENTLY`.

#### 1.4 — Resolve memory + swap pressure on the droplet

The droplet log earlier showed **Memory 84%, Swap 99%**. Swap thrashing
makes everything slower under load — Postgres, Redis, and Python all
suffer. Three envs sharing one droplet is the root cause; you can't
optimize your way out of it past a point.

- Short-term: reduce per-env resource limits in the `deploy.resources`
  blocks of the three compose files. Lower celery workers' memory ceiling.
  Add swap pressure alerts to catch this earlier.
- Medium-term: split prod onto its own droplet. Test + stage can stay
  together; production deserves dedicated resources.
- Effort: short-term 1-2 hours. Medium-term: 1-2 days including DNS
  changes, data restore drill, monitoring setup.

### Tier 2 — Worthwhile after Tier 1

#### 2.1 — Eager-load product relations (kill N+1 queries)

If any of the product/cart/promotion serializers fetch relations lazily
(`product.category`, `product.images`, `cart.items[*].variant.product`),
each VU iteration multiplies that into N+1 queries. SQLAlchemy:
`selectinload()` or `joinedload()` to fetch in one round-trip.

- Trace with SQLAlchemy's `echo=True` in dev under a single test request
  and count the number of SELECTs. If it's >5 for a "get one product",
  there are N+1s.
- Effort: 2-4 hours per offender. Be careful — over-eager loading bloats
  the response. `selectinload` for collections, `joinedload` for 1-1.

#### 2.2 — Move analytics/tracking writes off the request path

Storefront events (page_view, add_to_cart, etc.) currently write
synchronously inside the request handler. At 100 VUs that's 100×
write-locks on the events table competing with reads.

- Push events to a Celery queue (you already have celery-worker).
- The request returns immediately; the worker drains the queue in
  batches.
- Effort: 3-5 hours including idempotency design for retried events.

#### 2.3 — PgBouncer in front of Postgres

FastAPI's async pool opens a new Postgres connection per VU at scale.
PgBouncer in transaction mode pools connections at the layer above, so
your API can have 100 logical connections backed by 10 real ones.

- Add a `pgbouncer` service to the compose stack.
- Point `DATABASE_URL` at port 6432 instead of 5432.
- Set transaction-mode pooling (most Python ORMs are compatible; verify
  no `LISTEN`/advisory locks in your code).
- Effort: 2-3 hours. Big win at higher concurrency, modest at 100 VUs.

#### 2.4 — Cloudflare APO or aggressive page-level caching for the storefront

The Next.js storefront SSRs every request. For un-personalized pages
(home, PLP, PDP) with the same query, the rendered HTML is identical for
all shoppers. Cloudflare APO (or a manual edge-cache rule) can serve the
same HTML for the TTL window.

- Set up a Cache Rule: cache HTML for hostnames matching `*.numueg.app`,
  bypass for paths matching `/(api|account|cart|checkout)`, TTL 5-15 min.
- Effort: 1-2 hours. Massive impact on cold-cache home traffic.

### Tier 3 — Bigger investments

#### 3.1 — ISR / static generation for product pages

Next.js supports per-page ISR — render once, serve from disk, revalidate
on a schedule. PDP pages are a strong candidate: shoppers re-load the
same product many times, and inventory updates can drive on-demand
revalidation via webhook.

- Effort: 1-2 days including invalidation wiring. Real win is taking
  PDP entirely off the SSR hot path.

#### 3.2 — Read replica for Postgres

If after all the above you're still I/O bound on Postgres, a replica
takes read pressure off the primary.

- DO Managed Postgres supports replicas natively.
- Route read-only queries (product listing, search, theme, promotions)
  to the replica via a separate SQLAlchemy engine + session factory.
- Effort: 1-2 days including the read/write split plumbing.

#### 3.3 — Horizontal API scaling

One API container per env today. Run 2-4 behind nginx with sticky
sessions disabled (no in-process state to break).

- Cleaner version: real load balancer (Cloudflare → multiple droplets).
- Required for any meaningful "more than 100 VUs" capacity.
- Effort: depends on infrastructure direction.

### What I'd skip

- **GraphQL or sparse-field selection** — your endpoints are already
  per-purpose, not over-fetching. Adding a query layer would be cost
  without benefit at current scale.
- **Custom CDN per asset class** — Cloudflare in front of the droplet is
  enough. R2 is already CDN-fronted for product images.
- **Premature horizontal sharding of Postgres** — the dataset is nowhere
  near that scale; the wins are all in caching + indexes first.

## Suggested investment order (concrete)

If you have one afternoon: **1.2 (cache headers) + 1.1 (Redis store/theme cache)**.

If you have one day: above + **1.3 (indexes) + 2.4 (Cloudflare HTML cache)**.

If you have a week: above + **2.1 (N+1) + 2.2 (async tracking) + 2.3 (PgBouncer)**.

If you have two weeks: also **3.1 (ISR for PDP) + 1.4 medium-term (split prod droplet)**.
